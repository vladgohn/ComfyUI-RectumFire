# RectumFire Growth Roadmap

This file is the execution plan for turning `ComfyUI-RectumFire` from a useful node pack into an income-generating project.

The goal is not generic "promotion."
The goal is the shortest practical path from:

- useful open-source node pack

to:

- discoverability
- trust
- inbound interest
- first paid support
- recurring support

## Success Criteria

Short-term success:
- package is easy to install and easy to understand
- package is visible where ComfyUI users actually discover tools
- package has a clear support link and clear paid offer

Mid-term success:
- first sponsor(s)
- first paid workflow repair / cleanup request
- repeatable showcase content

Long-term success:
- recurring sponsor income
- repeat inbound consulting / workflow work
- reputation in the ComfyUI utility / workflow UX niche

## Positioning

Primary positioning:

`Workflow repair, observability, and daily-use utilities for ComfyUI.`

Paid positioning:

`I fix broken ComfyUI workflows, improve graph UX, and build utility nodes for real production use.`

This project should be sold on solved pain, not on "more nodes."

## Priority Order

1. Distribution
2. Conversion surface
3. Trust assets
4. Paid offer
5. Recurring support

That means:
- first make it easy to discover and install
- then make it easy to support or hire you
- only after that scale content and memberships

## Phase 1: Distribution Foundation

Goal:
make RectumFire discoverable through real ComfyUI user flows, not only through manual GitHub traffic.

### Task 1.1: Prepare package for Comfy ecosystem distribution

Target:
- repository metadata cleaned up
- install path clear
- package structure ready for registry submission

Likely repo work:
- add package metadata files required by current Comfy registry flow
- add clean project description
- add tags / topics / contact links
- verify install docs and exported node list

Owner:
- Codex prepares files
- user handles any external submission/auth steps

Definition of done:
- repo is technically ready for registry submission
- no placeholder text remains

### Task 1.2: Submit to Comfy Registry / ensure Manager discoverability

Why this is high priority:
- this is the shortest path to installs from actual ComfyUI users
- users search from Manager, not only from GitHub

Owner:
- Codex prepares submission material
- user completes auth / submission if needed

Definition of done:
- package is submitted or all submission assets are ready

## Phase 2: Conversion Surface

Goal:
turn passive GitHub visitors into supporters or leads.

### Task 2.1: Enable GitHub Sponsors

Why first:
- lowest friction support path for open-source
- best fit for GitHub-native discovery
- better first monetization rail than Patreon

Owner:
- user creates/enables account-side settings
- Codex writes sponsor text, tier copy, and profile copy

Definition of done:
- Sponsor button is live
- 3 simple tiers exist
- sponsor profile text is ready

Suggested initial tiers:
- $3-5: support the project
- $10-15: early access / roadmap voting / dev updates
- $25-50: priority issue review or workflow office-hours note

Important:
do not overpromise recurring deliverables early.

### Task 2.2: Add support CTA in repository

Required repo assets:
- short support section in README
- `FUNDING.yml` if using GitHub Sponsors
- one short "hire me" / "support the project" block

Definition of done:
- every interested user can find how to support you in under 10 seconds

## Phase 3: Trust Assets

Goal:
prove value fast with concrete examples.

### Task 3.1: Build 3 showcase workflows

These are not generic demos.
They should prove solved pain.

Workflow A:
- Broken Workflow Recovery
- uses `Fire Resolve + Fire Copy + Fire Note`

Workflow B:
- Subgraph Preview / Observability
- uses `Fire Banner`

Workflow C:
- Clean Routing + Run Feedback
- uses `Fire Switch + Fire Timer + Fire Done`

Definition of done:
- each workflow is shareable
- each workflow has 1 image and 1 short clip/gif
- each workflow has one-sentence value proposition

### Task 3.2: Create concise demo assets

Required:
- before / after screenshots
- 10-25 second clips
- no long tutorial needed at first

Purpose:
- social proof
- Civitai listing content
- GitHub visual proof
- sponsor/profile proof

## Phase 4: Paid Offer

Goal:
get the first direct payment faster than waiting for community donations alone.

### Task 4.1: Define the paid offer

Primary offer:
- ComfyUI workflow repair and cleanup

Secondary offers:
- workflow UX cleanup
- subgraph refactor and observability setup
- custom utility node work

Offer format:
- fixed-scope audit / repair
- not vague consulting

Recommended initial packaging:
- Workflow Repair: one broken workflow, one pass, one delivery
- Workflow Cleanup: remove clutter, improve routing, improve maintainability
- Subgraph UX Pass: add observability and external preview strategy

Definition of done:
- one short services page exists
- one DM/email CTA exists
- pricing can be discussed privately at first

## Phase 5: Publishing Pipeline

Goal:
put RectumFire where ComfyUI users already look.

### Task 5.1: Publish showcase workflows and posts

Priority channels:
- Civitai
- GitHub releases
- ComfyUI-oriented communities

Content angle:
- not "new nodes released"
- instead "I fixed this painful workflow problem"

Publishing topics:
- repairing imported workflows automatically
- making subgraphs visually observable
- cleaning up branch switching in large graphs

Definition of done:
- at least 3 public posts/assets exist

### Task 5.2: Light community presence

This is not the first step.
It only works after the assets and offer exist.

Use for:
- distribution
- credibility
- funneling to repo/support/services

## Phase 6: Patreon Only After Signal

Goal:
launch recurring membership only after there is evidence of audience pull.

Trigger to launch Patreon:
- at least first sponsors or first paying users exist
- at least one content rhythm is realistic to sustain

What Patreon should sell:
- monthly utility drops
- early node access
- workflow packs
- roadmap influence

What Patreon should not be:
- the first monetization experiment
